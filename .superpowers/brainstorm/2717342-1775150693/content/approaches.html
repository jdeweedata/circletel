Three Approaches — Fix Bugs + Reduce Friction

All three fix the critical issues. They differ in how much the flow is restructured.

A

Surgical Fixes — Patch in Place

Keep all 5 screens. Fix bugs. Merge one redundant page.

5 → 4 screens Lowest risk ~1 week
✓ OTP: Defer phone verification to post-order (send code, verify later in dashboard)
✓ Confirmation: Add API fetch by orderId, remove localStorage dependency
✓ Packages: Wire to real /api/coverage/packages?leadId= API
✓ Duplicate orders: Consolidate to payment stage only (remove from service-address)
✓ Collapse service-address: Merge property type dropdown into coverage page (step 1)
→ Flow becomes: Coverage+PropertyType → Packages → Account → Payment → Confirm

Pros

  • Minimal blast radius
  • Reuses existing pages
  • Fast to ship

Cons

  • Still 4 screens
  • Account page still heavy
  • OTP deferred ≠ removed
B

3-Step Streamlined Flow (Recommended)

Restructure into 3 lean steps. Remove detours entirely.

6 → 3 screens Best conversion ~2–3 weeks
1
Where are you? — Address autocomplete + property type (house/apt/estate) + coverage type in one screen
2
Which plan? — Real packages from API. Select plan + optional add-ons.
3
Who are you + pay — Email/Google signup, name/phone, inline order creation + R1.00 payment. One screen, two sub-sections.
✓ No OTP in flow — phone collected, verified post-order in dashboard
✓ Order created once, at payment — no duplicates possible
✓ Confirmation fetches from API — survives refresh
✓ Progress bar stays honest: 3 real steps

Pros

  • Fewer drop-off points
  • Cleaner mental model
  • Fixes all critical bugs

Cons

  • Rewrites 3 pages
  • Step 3 is busier
  • More testing needed
C

Single-Page Checkout

Everything on one scrollable page. Sections reveal progressively.

1 long page Highest risk ~4–6 weeks
Address → packages appear below → account section reveals → payment at bottom
Sticky order summary sidebar (like Shopify checkout)
No page navigation, no state loss on back-button
⚠️ Large architectural change — rewrites everything
⚠️ Mobile UX requires careful design (long scroll)

Pros

  • No back-button state loss
  • Amazon/Shopify-familiar
  • Fewest round trips

Cons

  • Full rewrite
  • Complex mobile UX
  • Months of work
