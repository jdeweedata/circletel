# Standalone Payload CMS App Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deploy a working, clickable Payload CMS admin at `https://cms.circletel.co.za`, backed by the existing Supabase Postgres `payload` schema, without touching the main CircleTel Next.js app.

**Architecture:** A **separate, minimal Next.js 15 + Payload 3.x app** (its own directory `/home/circletel-cms`, its own Git repo, its own Coolify container). It reuses the existing `payload.config.ts` + 5 collections verbatim and connects to the **same** Supabase Postgres (`schemaName: 'payload'`). Because it is its own app, Payload's `RootLayout` owns the document `<html>` — so the admin UI hydrates correctly (this is exactly what the in-monolith `/cms` could not do without restructuring all ~254 routes). The main app is untouched; if it ever needs CMS content it reads the same `payload` schema via the Local API or REST.

**Tech Stack:** Next.js 15, Payload 3.85, `@payloadcms/db-postgres`, `@payloadcms/next`, `@payloadcms/richtext-lexical`, `@payloadcms/plugin-seo`, Docker (node:20-alpine), Coolify on VPS 94.72.104.81, Traefik, Let's Encrypt.

**Why this over in-monolith `/cms`:** The monolith has a single root `app/layout.tsx` rendering `<html>`. Payload's admin needs its own `<html>` root layout; two root layouts require removing `app/layout.tsx` and moving ~all routes into a route group (high blast radius on a near-OOM 254-page build). A standalone app sidesteps the conflict entirely — same pattern already used for the Vue admin (`admin.circletel.co.za`).

| File | Responsibility |
|------|----------------|
| `package.json` | Deps + scripts (`dev`, `build`, `start`, `generate:types`) |
| `next.config.mjs` | `withPayload()` wrapper + `output: 'standalone'` |

| `.dockerignore` | Exclude node_modules/.next/.git |
| `.env.example` | Document required env (no secrets) |

```bash
cat > /home/circletel-cms/Dockerfile <<'DOCKER'
# ---- deps ----
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json* ./
RUN npm ci || npm install --legacy-peer-deps

# ---- builder ----
FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
# DATABASE_URL/PAYLOAD_SECRET are provided at build via --build-arg or Coolify build env
ARG DATABASE_URL
ARG PAYLOAD_SECRET
ENV DATABASE_URL=$DATABASE_URL PAYLOAD_SECRET=$PAYLOAD_SECRET NEXT_TELEMETRY_DISABLED=1
RUN npx payload generate:importmap && npm run build

# ---- runner ----
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production NEXT_TELEMETRY_DISABLED=1 PORT=3015 HOSTNAME="0.0.0.0"
RUN addgroup --system --gid 1001 nodejs && adduser --system --uid 1001 nextjs
COPY --from=builder /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
USER nextjs
EXPOSE 3015
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=5 \
  CMD wget -qO- http://127.0.0.1:3015/api/health || exit 1
CMD ["node", "server.js"]
DOCKER
```
