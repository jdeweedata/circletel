## ğŸ” Access Control

### Required Permission
- `PERMISSIONS.PARTNERS.VIEW` - Required for all partner portal pages

### Authentication Flow
1. User logs in via Supabase Auth
2. Middleware checks auth cookie
3. PermissionGate component checks partner permission
4. If denied: Redirect to home or show "Access Denied"
5. If approved: Load partner portal

### Partner Status
- **Incomplete:** Registration started, not submitted
- **Submitted:** Awaiting admin review
- **Under Review:** Admin reviewing documents
- **Verified:** Approved and active
- **Rejected:** Not approved
- **Suspended:** Temporarily disabled

## ğŸ“Š Data Flow

### Registration â†’ Onboarding
1. User fills registration form
2. POST `/api/partners/onboarding`
3. Creates partner record (status: incomplete)
4. Redirects to `/partners/onboarding/verify`

### Document Upload â†’ Compliance
1. User uploads files via form
2. POST `/api/partners/compliance/upload`
3. Stores in Supabase Storage bucket
4. Creates document record
5. Updates progress percentage
6. When all required uploaded â†’ Enable submit
7. POST `/api/partners/compliance/submit`
8. Updates status to 'submitted'

### Admin Review â†’ Activation
1. Admin reviews in backend
2. Verifies each document
3. Updates partner status to 'verified'
4. Assigns partner number (CTPL-2025-XXX)
5. Sets tier and commission rate
6. Partner gains full portal access

### Lead Management â†’ Activities
1. Admin assigns lead to partner
2. Partner views in `/partners/leads`
3. Partner opens lead detail
4. POST `/api/partners/leads/[id]/activities`
5. Creates activity record
6. Updates lead timestamps

### Order Conversion â†’ Commission
1. Customer order created
2. System detects assigned partner
3. Calls `create_tiered_commission()` or `create_margin_commission()`
4. Creates transaction (status: pending)
5. Admin reviews and approves
6. Status â†’ approved â†’ paid
